# Capability phases

Phases are ordered by semantic dependency, not calendar. A phase begins when its entry conditions hold and completes only when every exit gate has retained evidence.

| Phase | Outcome |
| --- | --- |
| [00](00-foundation.md) | Contracts, schemas, fixtures, logs, plugin, and repository integrity |
| [01](01-baseline-analysis.md) | Reproducible baseline compilation and loop classification |
| [02](02-counterfactual-search.md) | Isolated noalias variants and honest search minimality |
| [03](03-obligation-derivation.md) | Supported source-level obligations and typed declines |
| [04](04-agent-workflow.md) | Repository-aware Codex repair and refusal workflow |
| [05](05-validation-benchmarking.md) | Behavior, compiler, sanitizer, and performance evidence |
| [06](06-product-hardening.md) | Distribution, security, compatibility, and coherent UX |
| [07](07-expansion.md) | Additional counterfactual families and toolchains |

Each phase document defines entry conditions, deliverables, adversarial cases, and exit gates. [PLAN.md](../../PLAN.md) is the single status index.
